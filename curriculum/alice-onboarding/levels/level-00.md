# Level 0: The First File

## Learning Objective

After completing this level, the learner will have:
> Created a project folder named after something they want to accomplish, with a PRIMER.md inside it that describes the project — what it is, what's active, and what's next.

**Why this matters:** Level 0 is the zero-threshold entry. No account. No installation. No keys. No sign-up. One question: "What do you want to accomplish?" One folder. One file. If the learner can do this, they are already doing koad:io — even if they don't know what koad:io is yet. The pattern is: one folder per project, every folder gets its own PRIMER.md. The entire curriculum is a series of decisions about how much deeper to go from here.

---

## Knowledge Atoms

### Atom 0.1: What Do You Want to Accomplish?

**Teaches:** The starting point is not a file or a folder — it's a question. What are you trying to do?

Before we talk about files or folders or any technical thing — what do you want to accomplish?

Something you've been meaning to get to. A project you keep thinking about. A goal you haven't started yet, or one you've started but it's scattered across your head and your browser tabs and your notes app.

Name it. That's where we start.

---

### Atom 0.2: One Folder Per Project

**Teaches:** The fundamental organizing rule — every project gets its own folder, and every folder IS a project.

That thing you want to accomplish? It gets a folder. Just one. Named after the thing.

This is the rule: **one folder per project.** Not "my stuff." Not "documents." Not "misc." One folder, one purpose. The name of the folder is the name of what you're trying to do.

Where to put it depends on your machine:
- **Windows:** Your Desktop is fine. Or Documents. Wherever you'll see it.
- **Mac:** Desktop or Documents. Same idea.
- **Linux:** Your home directory, or Desktop.

Even right on the desktop is fine. We can organize later. The important thing is that it exists — one place for one project.

If you have five projects, you'll have five folders. Each one knows what it is because each one has its own description file inside it.

---

### Atom 0.3: PRIMER.md — The Project Describes Itself

**Teaches:** What a PRIMER.md is, how it works, and the immediate payoff.

Inside your new project folder, create a file called PRIMER.md. This is a plain text file where you describe the project. Three things:

1. **What is this project?** One or two sentences.
2. **What are you focused on right now?** The current thing.
3. **What's next?** The thing after that.

That's it. Here's an example:

```markdown
# Home Renovation

Planning and tracking the kitchen renovation.
Current focus: getting contractor quotes.
Next: compare quotes and pick materials.
```

When any AI session opens this folder, it reads the PRIMER.md and arrives oriented. No re-explaining. No "let me catch you up." The project describes itself.

Every project folder gets its own PRIMER.md. The folder is the project. The PRIMER.md is the project describing itself. One folder, one file, one purpose.

---

### Atom 0.4: This Is the Pattern, Not a Workaround

**Teaches:** Where the folder-per-project pattern fits in the larger journey — and that it's a legitimate destination, not just a stepping stone.

A project folder with a PRIMER.md is not a trial version of something more complex. It is a real, working unit. Journalists, researchers, developers, and operators all use this pattern — a folder that knows what it is because it contains its own description.

But it is also the first step of a path, if you want to walk it:

```
project folder       → add PRIMER.md      → project that describes itself   [5 minutes]
described project    → add hooks/         → project that can do things      [1 hour]
project with hooks   → add keys + .env    → sovereign entity               [half day]
sovereign entity     → add trust bonds    → authorized network participant  [ongoing]
```

Every step is optional. Every step is additive. Nothing breaks at any transition.

You can stop at one folder with one PRIMER.md and get value forever. Or you can keep walking. The curriculum you're in now is about that walk — but the walk starts here, with one project, one folder, one file.

### Opening

**Alice:** Before we get into anything technical — I want to ask you a simple question.

What do you want to accomplish? Not in general. Something specific. A project you've been meaning to start, or one you've already started but it's scattered — in your head, in browser tabs, in notes you can't find.

What's the thing?

**Alice:** Good. That's a project. And every project deserves its own place to live.

Can you do that right now?

3. What's next?

```markdown
# Kitchen Renovation

```

Tell me what you wrote — even just roughly.

**Human:** [describes what they wrote]

### Exchange 3

**Alice:** That's the whole thing. You now have a project that describes itself.

Here's why that matters: the next time you open any AI conversation and point it at that folder — the AI reads your PRIMER.md and arrives already knowing what you're working on. No re-explaining. No "let me catch you up." The project told it.

And here's the pattern that makes it powerful: **one folder per project. Every folder gets its own PRIMER.md.** Five projects? Five folders, five PRIMER.md files. Each one describes its own thing. Each one is self-contained.

The folder IS the project. The PRIMER.md IS the project describing itself.

**Human:** That's... actually really simple.

**Alice:** It is. And it doesn't stop being useful. You wrote it once; it works forever. You can update it as the project moves. You can come back to it in two weeks and know exactly where you left off — and so can any AI you bring in to help.

---

### Exchange 4

**Alice:** I want to tell you something about what you just did. It's not a workaround or a trick. It's the first step of a real pattern that scales all the way up — to projects that can do things automatically, to folders with their own identity and keys, to infrastructure that runs while you sleep.

Most people never go past this. They use one folder, one PRIMER.md per project, and that's the whole story. That's completely fine. The project describes itself. The work continues without re-explaining.

Some people keep walking. That's what this curriculum is about — if you want to.

But right now, you already did the thing. You have a project with a place to live and a file that describes it. That's not nothing.

---

### Landing

**Alice:** Here's the short version. You had a project that lived in your head — or scattered across tabs and notes and half-remembered plans. Now it lives in one folder, described in one file, readable by anyone who walks in — human or AI.

You wrote it once. It doesn't expire. No service is holding it. If you moved it to a new machine, it would still say exactly the same thing, because it's just a file on your disk.

That's the idea behind everything we're going to cover. Not in the abstract — we just did it.

Ready to go deeper?

---

### Bridge to Level 1

**Alice:** So: you have a project folder that knows what it is. What IS that folder, actually? It's a directory. And when you start adding the right structure to a directory — keys, configuration, commands, a specific shape — it becomes something more: an entity. That's what koad:io is built from. Let's look at what that means.

---

### Branching Paths

#### "I already do something like this with README files"

**Human:** I already put README files in my projects. This seems like the same thing.

**Alice:** You're right — it's the same instinct. Describe the thing so anyone arriving later knows what they're looking at. The difference is subtle: a README is usually for humans, written once, documents the finished shape. A PRIMER.md is for agents as much as humans — it's the current state, the active context, the "what's happening right now." It gets updated as the project moves. Less "documentation," more "orientation." You can absolutely rename your existing README to PRIMER.md and it works fine. The practice is the same one you've already built.

---

#### "What if I don't use the terminal?"

**Human:** I don't really use the terminal. Can I still do this?

**Alice:** Completely. You don't need a terminal at all. Create a folder the normal way — right-click, "New Folder." Open any text editor — Notepad, TextEdit, VS Code, whatever you use. Create a new file, name it PRIMER.md, write your three lines, save it in the folder. That's it. The file is just text. The .md extension just means it's formatted as markdown, which is a simple way of writing that both humans and machines can read.

---

#### "I have too many projects — where do I start?"

**Human:** I have like ten things going on. Which one do I pick?

**Alice:** Pick the one that's weighing on you most. Not the biggest one — the one you think about when you're trying to fall asleep. That's the one that needs a place to live first. The others will get their folders too — one at a time. But start with the one that's taking up the most space in your head.

---

#### "What should I actually write in it?"

**Human:** I'm not sure what to write. What's the right format?

**Alice:** There isn't one. Whatever you'd want to know if you came back to this project after two weeks away, or whatever you'd tell someone walking in cold. A few prompts that help:
- What is this folder for?
- What are you working on right now?
- What's blocked, and what's next?
- Are there any files in here someone should know about?

You don't need to answer all of those. Even one sentence — "this is my notes on X, currently focused on Y" — is enough to orient an agent. Start minimal. Add more when you feel like something's missing. The PRIMER.md will tell you what it needs if you use it — you'll notice when an agent asks about something that should have been in there.

## Exit Criteria

The learner has completed this level when they can:
- [ ] State what they want to accomplish (the project they chose)
- [ ] Explain the one-folder-per-project rule (every project gets its own folder, every folder gets its own PRIMER.md)
- [ ] Have created a project folder with a PRIMER.md inside it
- [ ] Describe what the PRIMER.md does (the project describes itself so any human or AI arriving later knows what's happening)

**How Alice verifies:** The learner should be able to say what project they chose, where the folder lives, and what they wrote in the PRIMER.md. The goal is not conceptual precision — it is a lived action. Did they make the folder? Did they write the file? Do they understand that every project gets this treatment? That's the gate.

---

## Assessment

**Question:** "If you come back to this folder in two weeks, or send an AI into it cold — what will it know about your project?"

**Acceptable answers:**
- "It'll know what I'm trying to do and where I left off."
- "The project describes itself — the PRIMER.md tells it what's happening."
- "I don't have to re-explain. The context is in the file."

**Red flag answers (indicates level should be revisited):**
- "I'm not sure" — learner has not completed the action
- "It would know everything automatically" — overclaiming; PRIMER.md orients, it doesn't magically understand every detail
- Confusion about needing an account or service to use PRIMER.md
- Treating PRIMER.md as a configuration file rather than a description

**Estimated conversation length:** 4–8 exchanges

## Alice's Delivery Notes

This is the zero-threshold level. The learner may not know what koad:io is. They may not be sure why they're here.

**The job of Level 0 is:**
1. Ask "What do you want to accomplish?" — start from the project, not the file system.
2. Get them to make one folder named after that project.
3. Get them to write a PRIMER.md inside it.
4. Establish the pattern: one folder per project, every folder describes itself.

**Lead with the question, not the file system.** If a learner is uncomfortable with folders, don't teach folder concepts — ask what they want to accomplish and work backward from there. "What do you want to do?" → "That's a project." → "Let's give it a home." → "Now let's describe it."

**Platform guidance:** Be specific about WHERE to create the folder. Desktop is fine for any platform. Don't assume terminal comfort. Right-click → New Folder is a valid path. Notepad/TextEdit is a valid editor.

**If they have many projects:** "Pick the one that's weighing on you most. The others will get their folders too — start with the one taking up the most space in your head."

The emotional tone is: *this is easy, it works right now, you've already done it.* Not a pitch. Not a preview of what's coming. Just a useful thing that works immediately.

Do not mention trust bonds, daemon, peer rings, or the entity stack in this level. Do not mention gestation, keys, or GitHub. The entire world of Level 0 is: what do you want to accomplish? Here's where it lives. Here's how to describe it.

The bridge to Level 1 names the word "entity" for the first time. That's the transition. Until that bridge, the curriculum lives in the learner's existing vocabulary — projects, folders, files, text editors. Nothing alien.

If a learner asks "is this the whole thing?" — the answer is: "For today, yes. You can do more, but you don't have to. Ready to see what more looks like?"
